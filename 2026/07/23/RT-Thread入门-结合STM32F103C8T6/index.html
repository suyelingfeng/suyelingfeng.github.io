RT-Thread 入门：结合 STM32F103C8T6 理解线程与设备

发表于2026-07-23|更新于2026-07-23|嵌入式
|阅读量:

最早写单片机程序时，我习惯把所有逻辑都塞进 while (1)：轮询按键、读取传感器、刷新屏幕，再顺手处理串口。功能少的时候很直接，功能一多，某个环节稍微阻塞，其他事情就会一起受影响。

RT-Thread 解决的并不是“让 MCU 同时执行很多代码”，而是把 CPU 时间交给内核管理。我们把不同工作拆成线程，再用优先级、等待和线程间通信描述它们之间的关系。这样做会增加一点学习成本，但项目结构会清楚很多。

本文使用的是常见的 STM32F103C8T6。有些资料会误写成 STM32F203C8T6；ST 的正式型号中，蓝板常用芯片是 STM32F103C8T6。

我目前也在学习 RT-Thread。这篇文章不是一份已经完成的项目总结，而是我结合 STM32F103C8T6 整理的阶段性认识。后续会按照文末的练习顺序逐项动手验证，再把遇到的问题和实际结果补充进来。

1. 这块 MCU 适合跑什么版本

STM32F103C8T6 使用 Cortex-M3 内核，最高主频 72 MHz，标称 64 KB Flash、20 KB SRAM。这个配置能运行 RT-Thread 标准版，但空间并不宽裕，不适合一开始就打开文件系统、网络协议栈和大量软件包。

刚入门可以这样选：

目标 建议
只需要线程、信号量、邮箱等内核能力 RT-Thread Nano
希望使用 FinSH、设备框架和软件包 精简配置的 RT-Thread 标准版
只是验证调度与线程 API 先用官方 STM32F103 工程或 RT-Thread Studio 创建最小工程

Nano 不是另一套 API，而是从标准版中裁剪出的轻量内核。对于 20 KB SRAM 的 F103C8T6，线程栈和动态堆都要精打细算。

2. RT-Thread 不只有一个调度器

RT-Thread 的核心包括线程管理与调度、时钟、定时器、内存管理以及线程间通信。再往上还有设备框架、FinSH 命令行、文件系统和软件包等组件；BSP 则负责把内核接到具体芯片和开发板上。

这张图里最值得注意的是设备框架。应用通常不必直接调用某个 UART 驱动的私有函数，而是按设备名查找，再通过统一接口访问。以后更换串口实例或底层驱动时，上层业务代码不需要跟着大改。

3. 线程是怎样获得 CPU 的

RT-Thread 使用基于优先级的抢占式调度。数值更小的优先级通常代表更高优先级；高优先级线程一旦就绪，可以抢占当前的低优先级线程。

一个线程大致会经历这些状态：

  1. 创建后处于初始状态；
  2. 调用 rt_thread_startup() 后进入就绪队列；
  3. 调度器选择它时开始运行；
  4. 延时或等待资源时进入挂起状态；
  5. 等待结束后重新就绪，线程结束后进入关闭状态。

“实时”不等于所有线程都跑得快，而是关键工作能在可预期的时间内得到响应。把每个线程都设成高优先级，反而会让系统难以分析。

在一个简单的环境监测项目中，可以先这样分工：

线程 工作 优先级思路
sensor 定时读取传感器 较高，读取后立即让出 CPU
console 处理串口命令 中等，大部分时间阻塞等待
led 闪烁状态灯 较低，不影响核心功能

优先级不是按“功能重要程度”随手排列的，还要看响应期限、执行时间和线程是否可能阻塞。

下面的例子创建一个 LED 线程。引脚编号只是示例，实际工程要根据开发板原理图修改；不少 Blue Pill 板的板载 LED 接在 PC13，并且是低电平点亮。

rt_thread_create() 后面的几个参数分别是入口参数、栈大小、优先级和时间片。这里的 512 是字节数，够不够不能只凭感觉：局部数组、函数调用深度、格式化输出和浮点运算都会增加栈占用。

rt_thread_mdelay() 会让线程挂起一段时间，CPU 可以去执行其他就绪线程。它和占着 CPU 空转的延时循环不是一回事。

INIT_APP_EXPORT() 会把初始化函数放入 RT-Thread 的自动初始化序列。系统启动后会调用它，因此不需要再从 main() 手动调用一次。若工程没有启用组件自动初始化，可以直接在 main() 中调用 led_thread_init()。

5. 线程之间不要靠全局变量“碰运气”

假设传感器线程产生数据，显示线程负责消费数据。只用一个全局变量和标志位，容易遇到覆盖、读写不同步或忙等问题。RT-Thread 已经提供了对应工具：

对象 适合解决的问题
信号量 通知某个事件发生，或记录有限资源数量
互斥量 保护共享资源，并处理优先级反转问题
事件集 等待一个或多个事件标志
邮箱 在线程间快速传递固定大小的值或指针
消息队列 传递固定长度的数据消息

例如，串口接收回调只负责释放信号量，解析线程等待信号量后再处理数据。这样中断中的工作量更小，解析逻辑也不会堵住中断。

互斥量和二值信号量看起来相似，但含义不同。共享 I²C 总线或日志输出应优先考虑互斥量；单纯通知“数据到达”才更适合信号量。

6. 设备框架如何落到 UART 上

设备名和支持的打开标志取决于 BSP 配置。若 uart1 not found，先不要怀疑应用代码，应该依次检查：

  1. menuconfig 或 Studio 中是否打开 UART 驱动；
  2. BSP 是否启用了对应串口实例；
  3. 引脚复用和时钟配置是否正确；
  4. FinSH 中执行 list_device 能否看到设备。

这种排查顺序比直接改寄存器更有效，因为它能先确认问题位于配置层、驱动层还是应用层。

7. 在小内存芯片上容易踩的坑

线程建得太多

每个线程都需要独立栈空间。F103C8T6 只有 20 KB SRAM，把一个功能拆成十几个线程，通常得不偿失。没有独立阻塞条件和响应期限的逻辑，可以放在同一个线程里。

在中断里调用会阻塞的接口

中断服务函数应尽快结束。不要在里面延时，也不要等待互斥量。需要处理的数据可以放进环形缓冲区，再用信号量或事件通知线程。

用延时掩盖同步问题

“加 10 ms 延时后就正常”往往只是改变了问题出现的概率。共享资源应通过互斥量保护，线程间的先后关系应通过 IPC 表达。

忽略优先级反转

高优先级线程等待低优先级线程占用的共享资源时，中优先级线程可能让等待进一步延长。RT-Thread 的互斥量具有优先级继承机制，所以保护独占资源时不要随意用二值信号量替代。

8. 接下来的学习实践

只阅读 API 很难真正理解调度和线程间通信。我准备以 STM32F103C8T6 为实验平台，从一个小而完整的工程开始，按下面的顺序学习：

  1. 跑通串口控制台和板载 LED，熟悉工程结构与 BSP 配置；
  2. 创建两个不同周期的线程，通过串口输出观察调度结果；
  3. 用信号量连接串口接收回调与解析线程，理解线程从挂起到就绪的过程；
  4. 接入一个 I²C 传感器，再用互斥量保护共享总线；
  5. 加入 FinSH 命令，查看线程、设备和内存的运行状态；
  6. 记录各线程的栈占用和响应时间，再根据实测结果调整栈大小与优先级。

这个顺序也是我接下来准备逐项完成的实践计划。等这些实验跑通后，我会继续整理串口通信、IPC 和传感器驱动的具体过程。对我来说，目标不只是让 RT-Thread “点亮一颗 LED”，而是逐步理解它怎样组织一个真实的嵌入式项目。
