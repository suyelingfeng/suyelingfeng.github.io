RK3588 在 x86 Ubuntu 上配置 ARM64 交叉编译环境

发表于2026-07-23|更新于2026-07-23|嵌入式 Linux
|阅读量:

最近需要把几个 C++ 程序部署到 RK3588。直接在板子上编译能跑，但速度慢，也不方便把环境固定下来，于是把交叉编译过程重新整理了一遍。

实际用下来可以分成两种情况：

  • 无第三方板端依赖：安装 Ubuntu 提供的 AArch64 交叉编译器即可。
  • 依赖 OpenCV、Rockchip MPP、RKNN 或其他板端库：除了编译器，还需要从目标板同步一份 sysroot。

本文编译的是运行在 RK3588 Linux 用户空间中的程序，因此工具链目标必须是 AArch64 GNU/Linux。aarch64-none-elf 面向裸机环境，不能用于链接目标板上的 glibc 和普通 Linux 动态库。

4. 为什么需要 sysroot

只使用交叉编译器，可以编译不依赖板端特殊库的程序。一旦工程需要下面这些内容，就应该配置 sysroot：

  • 目标板上的 OpenCV、FFmpeg、GStreamer；
  • Rockchip MPP、RGA、RKNN Runtime；
  • 板端特有的头文件和动态库；
  • 与目标系统匹配的 glibc、libstdc++；
  • 通过 find_package() 或 pkg-config 查找目标库。

sysroot 可以理解为目标板根文件系统在开发主机上的镜像。编译期间，头文件和库从 sysroot 查找，但 CMake、Ninja 等构建工具仍在 x86 主机上运行。

8. RK3588 CPU 编译参数

RK3588 采用 ARMv8-A 64 位 CPU。为了保持兼容性，通常不需要手动指定 -march；交叉编译器默认生成的 AArch64 程序即可运行。

如果确认程序只部署到 RK3588，可以按项目测试结果增加优化参数：

但需要注意：

  • 不要在 x86 主机交叉编译时使用 -march=native，它会针对 x86 主机而不是 RK3588；
  • 激进的 CPU 参数可能降低程序在其他 ARM64 平台上的兼容性；
  • 性能敏感代码应通过实际基准测试决定参数，而不是只看理论指令集；
  • 对第三方库使用不同架构参数前，要确认它们之间的 ABI 和运行环境兼容。

10.3 GLIBC_x.x not found

这通常表示编译时使用的 glibc 比目标板更新。检查目标板版本：

原则是：构建所用 glibc 版本不能高于实际目标系统可提供的版本。解决方法包括：

  • 使用目标板 sysroot；
  • 使用 BSP 附带的交叉编译器；
  • 在与目标板相同发行版的容器中构建；
  • 升级目标板镜像；
  • 谨慎评估静态链接，而不是直接把 glibc 强行静态化。

12. 发布前再检查一次

最后把容易漏掉的地方列在一起：

  1. 使用面向 Linux 用户空间的 AArch64 工具链；
  2. 用目标板根文件系统构建 sysroot；
  3. 让 CMake 和 pkg-config 只从目标 sysroot 查找头文件与库；
  4. 在部署前通过 file、readelf 和 ldd 检查架构与动态依赖。

只依赖标准库的小程序，装好 g++-aarch64-linux-gnu 就能开始。涉及 OpenCV、MPP、RGA 或 RKNN 后，真正麻烦的往往不是编译器，而是 sysroot 和目标镜像没有对齐。以后更换板端镜像时，也要记得同步更新这套构建环境。
